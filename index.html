01 · Introduce

What AUDIT.md is.

AUDIT.md is a single markdown file you hand to an LLM (Claude, GPT, Gemini, Copilot, Cursor, Antigravity, Windsurf, Zed — any model, any IDE). The model reads the spec, scans a repo, and writes a structured audit report: findings, severity, evidence, HITL banners, and a resumable state index.

It is not a script. It is a contract — rules the model must honor, output shape it must conform to, and behaviors it must refuse. Every behavior has a canonical § home, a stable rule_id, and a frozen birth fingerprint so the harness can detect drift.

The interesting part is not the prompt. It is how the prompt is maintained — a regression harness that blocks a merge whenever a previously-passing rule flips to fail.

Principles · in one breath
01 · Mirror reality.
Findings reference real files, real lines, real evidence. No fabrication, no evidence-free claims.
02 · Closed contracts.
Item schema, severity ladder, output layout — closed top-level keys. A drifted shape fails the harness.
03 · Resumable.
A STATE-INDEX plus atomic persistence means a halted audit picks up exactly where it stopped.
04 · Protected from weakening.
Five anti-gaming rules — the analyzer itself is forbidden from weakening them.
The drift problem

Large prompts rot the moment you edit them.

A 1,900-line prompt has dozens of behaviors, constraints, and contracts. Change one clause and you might silently weaken three others — especially under a different model or IDE. The only thing worse than a prompt that doesn't work is a prompt that used to work.

The usual loop ("ask the model, eyeball the diff, ship") has no memory and no regression net. This project is the opposite.

Silent regressions
Edit passes on Claude, breaks on GPT-4 or Gemini — and nobody notices for a week.
Prompt bloat
Every incident spawns a new paragraph; nothing gets deleted. Token cost creeps up.
Anti-gaming collapse
A "helpful" edit softens the rule that was holding the model honest.
The answer
Three tightly-coupled files, a per-rule pass ledger, a frozen birth fingerprint per rule, a 16-practice PE rubric, and an analyzer that can only propose — never apply.

The auditor surfaces high-risk findings behind a HITL banner before acting autonomously.

03 · Use it

Run it on any project.

AUDIT.md is model-agnostic and IDE-agnostic. Hand the spec to any reasonably capable LLM, point it at a repo, invoke one of three modes, and read the report. The spec handles the rest — state files, resumption, HITL banners, evidence redaction.

1

Attach the spec

Drop AUDIT.md into your IDE context (Cursor, Antigravity, Claude desktop, Copilot, Cline, Continue, Windsurf, Zed) — or paste it into a chat turn. Attach the repo root alongside.

~150 KB · one file · no deps
2

Pick a mode

First-pass, resume, or targeted scope. The spec reads the .audit/ folder at repo root and picks up where the last run stopped.

first-pass · resume · scoped
3

Read the report

Output lands in .audit/AUDIT-<date>.md. Findings are typed, evidenced, and redacted. HITL banners gate high-risk items.

frontmatter · findings · STATE-INDEX

04 · Fine-tune

How an edit gets through.

Every edit follows a seven-stage gauntlet. The analyzer can only propose — never apply. A human flips PROPOSED → ACCEPTED. Regression runs first, baseline mutations are atomic, and the PE rubric pressure-tests the result.

1
Scan capture
Run a fixture — or a real audit — record the output.
2
Analyzer
Read-only meta-prompt clusters drift and emits PROPOSED-EDITS.
3
Human review
Approve / reject / defer. X-rules never weaken.
4
Apply + refingerprint
Patch AUDIT.md, bump sha256, update companion pin.
5
Re-run fixtures
Especially F007 + F008; capture per-rule pass/fail.
6
Coverage-sweep
Integrity gate. Rule-survival invariant. PROBLEMS: none.
7
Promote baseline
Atomic merge via promote-baseline.sh.

What 80 is based on

16 practices × 5 points.

The ceiling is not fixed. max = 5 × Npractices. Today N is 16, so the ceiling is 80. Add a 17th practice and the ceiling becomes 85.

Score scale — uniform across all practices
5
Best-in-class. Fully realized, no edge cases miss it.
4
Strong. Minor documented or harmless exceptions.
3
Adequate. Notable gaps, but not actively misleading.
2
Weak. Measurable drift or rework in some runs.
1
Absent or anti-pattern. The prompt works against it.
Hard gate: a proposed edit may never drop a single practice by ≥2, and may never drop the TOTAL ratio by ≥0.05 vs. the prior baseline.

Keeping the loop current

The rubric is versioned and extensible.

When a new prompt-engineering practice emerges, add it to pe-best-practices.md with 1/3/5 anchors. Total jumps to 85; ratio drops until you earn the new points. The regression harness keeps you honest — gaming the rubric breaks a fixture cell.

How to add a practice & raise the ceiling

Six-step mechanism

  1. 1
    Qualify it. The candidate must appear in ≥2 vendor docs or ≥2 independent reproducible studies. Single-vendor advocacy is noise; multi-vendor convergence is signal.
  2. 2
    Append, never renumber. Add it to pe-best-practices.md with the next integer index (17, 18, …). baseline.json cells reference practices by integer — renumbering breaks the ledger.
  3. 3
    Write the 1/3/5 anchors. Anchor "5" to a best-in-class observable (not a vibe). "1" is the anti-pattern. "3" is adequate-with-gaps. Keep anchors measurable — "AUDIT.md contains a §X.y…" beats "the spec should be clear."
  4. 4
    Score honestly. First score is usually 2–3, not 5. That drop is the point — the ratio in baseline.json → pe_scores shows where the work is.
  5. 5
    Earn the points. Edit AUDIT.md to hit the 5-anchor. One fine-tune cycle per rubric change — the coverage-sweep + hard-gate invariant keeps you from gaming it.
  6. 6
    Update the ceiling. max = 5 × N_practices. 16 → 80. 17 → 85. 20 → 100. The scale grows with the field; your evidence grows with it.
Guardrail: hard gate — no single practice may drop ≥2 points vs. prior baseline; TOTAL ratio may not drop ≥0.05 across a cycle. Adding a new practice is allowed to dilute; regressing existing ones is not.
Candidate new practices (watchlist)

Four patterns under observation

Tracked across Anthropic, OpenAI, and Google docs + recent model cards. Not yet added to the rubric — each needs multi-vendor survival evidence before earning an integer index.

17 ·
Context-window awareness
Spec should degrade gracefully at 8k / 32k / 200k+ contexts. Front-load load-bearing rules; repeat critical invariants near anchors that models attend to.
18 ·
Reasoning-mode compatibility
Behavior under extended thinking / reasoning modes (o-series, Claude thinking) should not diverge from non-reasoning output. Spec must not inadvertently suppress or inflate chain-of-thought.
19 ·
Tool-use invariance
AUDIT.md outputs are identical whether the agent is told to use tools, runs standalone, or runs inside an orchestrator. No rule should rely on a specific tool being present.
20 ·
Groundedness & verifiability
Every claim in the audit output is traceable to a specific rule ID + file path + line range. Hallucinated findings fail the invariant at capture time.
